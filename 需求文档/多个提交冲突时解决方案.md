针对目标项目存在多个提交时的冲突处理问题，我提出以下解决方案：

### 解决方案：分步骤执行与冲突报告机制

1. **分步骤执行 cherry-pick**：将批量执行的 `git cherry-pick ${commitIds.join(" ")}` 改为逐个提交执行，即对每个提交 ID 单独执行一次 cherry-pick 命令。

2. **增强冲突信息与定位**：

   - 在执行每个提交的 cherry-pick 前，输出明确的提示信息，包括"正在同步提交：[提交 ID] - [提交信息]"
   - 当发生冲突时，明确指出具体是哪个提交（包含提交 ID 和提交信息）导致了冲突
   - 输出冲突文件路径和具体冲突内容的摘要信息

3. **冲突隔离与流程继续**：

   - 当某个提交发生冲突时，使用 `git cherry-pick --abort` 或 `git reset --hard HEAD` 恢复到冲突前的状态
   - 记录该冲突信息，但不中断整个同步流程
   - 继续尝试同步后续的其他提交

4. **统一冲突报告**：

   - 在所有提交处理完成后，生成一份完整的冲突报告
   - 报告中列出所有发生冲突的提交（包含 ID、提交信息和冲突类型）
   - 提供解决冲突的具体建议和步骤

5. **提交分组与状态管理**：
   - 为每个提交的同步操作创建独立的执行上下文
   - 在每个提交执行前记录当前分支状态，确保冲突处理后不会影响后续提交的执行环境

该方案可以解决当前报错信息不明确和中间冲突中断后续提交的问题，提高工具的可用性和用户体验。
