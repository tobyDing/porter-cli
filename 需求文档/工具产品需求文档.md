porter-ci 工具产品需求文档

## 功能描述

- porter-ci 是一个基于 git 实现跨项目代码功能同步的 ci 工具。
- porter-ci 可以实现跨项目的代码功能同步，例如：当一个项目的代码发生改变时， porter-ci 可以自动将该项目的代码同步到其他目标项目中。
- 实现同步的原理是依赖 git cherry-pick 命令，将一个项目的代码变更应用到其他项目中。遇到冲突，需要用户手动解决冲突

## 使用流程

- 用户在项目根目录下执行 `porter-ci` 命令，porter-ci 会自动读取项目的配置文件 `.porter-ci.config.json`。
- porter-ci 会根据配置文件中的信息，以当前项目的分支的代码变更为基准，将变更应用到其他项目中。
- 同步完成后， porter-ci 会提示用户同步成功。

## 实现流程

实现流程分为以下几个步骤：启动工具、读取并检查信息、同步代码

### 启动工具

- `porter-ci`启动时，进行必要的提示
  - 读取当前项目的名称、当前项目分支的名称。提示用户当前项目的名称、当前项目分支的名称，确认无误后继续。
  - 询问用户是否填写 porter-ci 配置文件
  - 询问是否启动同步
- 以上信息都确认无误后，开始同步代码。

### 读取并检查信息

- 读取当前项目的信息
  - 读取当前项目的名称、当前项目分支的名称。
  - 读取当前项目的自创建以来新增的 git commit 信息
- 检查当前项目的信息
  - 检查当前项目的分支自创建以来，是否有新增的提交。必须有新增的提交，否则会抛错提示用户。
- 读取项目的配置文件信息。
  - 配置文件为 .porter-ci.config.json, 且该文件需要放在项目的根目录下。参考 [示例配置文件](./.porter-ci.config.json)
  - 配置文件内容中包含同步的配置信息。
  - 配置文件内容必须是合法的 json 格式。
  - 配置信息包含以下字段：
    - `projectName``：项目的名称，用于在同步时提示用户。
    - `commit-id`：指定从当前项目当前分支哪次 commit 开始同步。如果不指定，默认从当前项目当前分支的创建开始新增的第一个 commit 开始同步。可选项。
    - `targetProjects`：要同步的目标项目列表，每个项目包含以下字段：
      - `projectName`：项目的名称，用于在同步时提示用户。必须指定。
      - `projectPath`：项目的路径，用于指定要同步的项目路径。必须指定。
      - `branch`：要同步的目标分支，必须指定。分支名称不应包含`master`、`test`等关键字，否则会抛错提示用户分支名称不对。
- 检查项目的配置文件信息。
  - 检查配置文件格式是否正确、必要字段是否存在
  - 检查当前项目的分支名称是否符合规范，是否包含`master`、`test`等关键字。
- 以上任意检查一旦不通过，抛错提示用户做出相应修改
  - 检查阶段抛错后，ci 工具退出并停止运行，不继续执行同步代码。
  - 用户需要重新执行 porter-ci 命令。
  - 以上`启动工具-读取并检查信息`过程，用户重复执行，直至用户信息全部校验通过，再进行下一步
- 检查通过后，提示用户检查通过，进行同步流程。

### 同步代码

- 把上一步读取到的新增的提交，根据配置文件中的配置信息，批量提交应用到目标项目中。
  - 当前项目的当前分支自创建以来，新增的提交都需要同步到其他项目中。
  - `git cherry-pick` 批量提交。
- 遇到冲突时，porter-ci 会提示用户具体哪个目标项目冲突，用户手动解决冲突。
- 解决冲突后，需要用户手动提交变更到目标项目中。
- 没有冲突时，porter-ci 会自动提交到目标项目中。
- 提交完成后，由用户手动校验代码，确认无误后，再手动合并到目标项目的分支中。
- porter-ci 只负责代码的同步，不负责代码的提交和推送远程分支。
- 同步完成后，提示用户同步完成。

## 技术栈

- 使用 node.js 实现工具的功能。且 node 版本需要 22 版本以上，建议使用 22.18.0 版本。
- 使用 nvm 工具管理 node 版本需要项目显式指定 node 版本，例如：在项目根目录下创建 .nvmrc 文件，文件内容为 22.18.0。
- 使用 pnpm 工具管理项目依赖
- 使用当前构建 cli 包流行的构建技术，打包构建代码。如 vite 等。
- 使用 typescript 实现代码的类型检查。
- 使用 eslint 工具检查代码质量。
- 使用 prettier 工具格式化代码。
- 使用 husky 工具在提交代码前，自动检查代码质量。
- 使用 git 命令行工具实现代码的同步。

## 项目规范

- package.json 中必须包含 `bin` 字段，指定 cli 命令的名称为 `porter-ci`。
- package.json 文件需要包含项目必要的信息，例如：项目名称、项目描述、项目版本、项目依赖、启动调试打包等脚本命令。保证开发者可以方便地开发、调试 cli 工具。
- 代码需要包含必要的注释、说明文档。方便开发者理解。
  - 注释需要包含必要的信息，例如：函数的参数、返回值、异常等。
  - 说明文档需要包含必要的信息，例如：使用流程、实现流程等。
